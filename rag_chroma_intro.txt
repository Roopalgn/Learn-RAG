# Chroma Vector Database Workflow – Complete Key Points

## 1. Purpose of Using a Vector Database

* A **vector database (like Chroma)** stores data as **embeddings (vectors)**.
* Embeddings contain **semantic meaning**, not just raw text.
* The main goal is **retrieval**, not just storage.
* Data stored in vector databases is usually used together with **Large Language Models (LLMs)**.
* Workflow idea:

  1. Store documents as embeddings.
  2. Retrieve relevant embeddings using similarity search.
  3. Provide retrieved information to an LLM.
  4. LLM generates the final answer.

---

# 2. Two Main Layers in the Workflow

## 1. Application Layer (Top Layer)

This represents the **main application that interacts with users**.

Components:

### Queries

* User input to the application.
* Usually **text questions**, but can also be:

  * voice
  * requests
  * commands
* Example:

  * “What is AI?”
  * “Summarize this document.”

### LLM Context Window

* The **Large Language Model processes retrieved data**.
* Uses embeddings and relevant documents to understand:

  * **intent**
  * **context**
  * **semantic meaning**

Functions:

* Interpret user query
* Use retrieved data as context
* Generate a response

### Answer / Response

* The **final output generated by the LLM**.
* Returned back to the user through the application.

---

# 3. Chroma Database Layer (Bottom Layer)

This is where **actual vector storage and retrieval happens**.

It consists of:

### Documents

* Raw **unstructured data** stored in the database.
* Usually:

  * text documents
  * PDFs
  * articles
  * notes
  * webpages

Important point:

* Documents are usually **split into smaller chunks** before storing.

Example:
Instead of storing a whole PDF, it is divided into:

* paragraph chunks
* sections
* sentences

This improves **retrieval accuracy**.

---

### Embeddings

* **Vector representations of documents**.
* Created using **embedding models** (neural networks).

Each document chunk has:

* its **text**
* its **vector embedding**

Example conceptually:

Document:
"Artificial intelligence is transforming industries"

Embedding:

```
[0.23, -0.91, 0.45, 0.12, ...]
```

Purpose:

* Allows **semantic similarity comparisons**.

---

### Vector Similarity Search

The database retrieves information based on **vector distance**.

Common distance metrics:

* Cosine similarity
* Euclidean distance
* Dot product

Meaning:
Instead of keyword matching:

Keyword search:

```
Query: "AI impact"
Matches exact words
```

Vector search:

```
Query embedding similar to
documents discussing AI influence
even if wording is different
```

This enables **semantic search**.

---

# 4. Query Processing

When a user asks a question:

Example:

```
User: "How does AI help healthcare?"
```

Step 1 — Query Input

* User sends a query to the application.

Step 2 — Generate Query Embedding

* The text query must be converted into **an embedding vector**.
* This is done using an **embedding model** (neural network).

Example tools:

* OpenAI embeddings
* sentence-transformers
* other embedding models

Important concept:
Vector databases **cannot understand raw text**, they only work with vectors.

---

# 5. Query Embedding Interaction with Database

After creating the query embedding:

The system performs **similarity search** against stored document embeddings.

Process:

1. Compare query vector with stored vectors
2. Find **most similar embeddings**
3. Retrieve corresponding documents

Result:

* Relevant document chunks are returned.

---

# 6. Passing Retrieved Data to the LLM

The retrieved information is sent to the **LLM context window**.

The LLM now receives:

1. User query
2. Retrieved documents
3. Context information

Example input to LLM:

```
Query: How does AI help healthcare?

Retrieved context:
- AI improves diagnostics
- AI helps analyze medical images
- AI assists in drug discovery
```

The LLM processes this information.

---

# 7. Role of the LLM

The LLM uses retrieved embeddings to:

* understand **context**
* interpret **intent**
* generate **coherent responses**

Possible operations:

* summarization
* explanation
* answering questions
* generating new text

The LLM may:

* fetch related information
* interpret query intent
* produce new text based on patterns it learned during training.

# 8. Final Output

The LLM generates the **final answer**.

Example:

User query:

```
How does AI help healthcare?
```

LLM output:

```
AI assists healthcare by improving diagnostics, analyzing medical images, and accelerating drug discovery.
```

The answer is returned through the **application layer**.

# 9. Full Process Flow (End-to-End)

### Step 1

User sends a **query** to the application.

### Step 2

Application converts query into an **embedding**.

### Step 3

Query embedding is sent to the **Chroma vector database**.

### Step 4

Database performs **similarity search** against stored embeddings.

### Step 5

Relevant documents are retrieved.

### Step 6

Documents are passed to the **LLM context window**.

### Step 7

LLM processes the query + retrieved context.

### Step 8

LLM generates a **response**.

### Step 9

Application returns the **final answer to the user**.

from chromadb.utils import embedding_functions 
default_ef = embedding_functions.DefaultEmbeddingFunction()
name = "Paulo"
emb = default_ef(name)
print(emb)

what does this do 

This code generates a **numerical embedding (vector representation)** for the text `"Paulo"` using Chroma’s default embedding model. 
`from chromadb.utils import embedding_functions` ->imports Chroma’s built-in embedding utilities. 

`default_ef = embedding_functions.DefaultEmbeddingFunction()`-> creates an embedding function object that internally uses the pretrained sentence-transformer model all-MiniLM-L6-v2. 

`name = "Paulo"`-> defines the input text that will be converted into a vector. 

`emb = default_ef(name)`-> sends the text `"Paulo"` to the embedding model which tokenizes it, processes it through the transformer network, and outputs a **384-dimension vector** that represents the semantic meaning of the text.

 `print(emb)->` prints this vector to the console as a long list of numbers. These numbers are the coordinates of `"Paulo"` in the model’s embedding space and are used by vector databases like Chroma to perform **semantic similarity search**, meaning texts with similar meanings will have vectors that are close to each other in this space.

croma_client = chromadb.PersistentClient(path="./db/chroma_persist")
This replaces chromadb.Client() from the previous code. A PersistentClient creates a database that is stored on disk instead of in memory, so the data remains saved even after the program stops. The path="./db/chroma_persist" tells Chroma to store the database files inside the folder db/chroma_persist in your project directory.
Effect of these changes: previously the database existed only while the script was running, but now the embeddings and documents are persisted on disk and can be reused in future runs of the program.
